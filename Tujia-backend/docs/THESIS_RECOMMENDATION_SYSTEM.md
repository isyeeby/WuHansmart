## 1 问题形式化

设民宿平台共有 \(N\) 套房源，房源集合记为 \(\mathcal{L}=\{l_1,l_2,\dots,l_N\}\)。对于任意用户 \(u\)，推荐系统的目标是从 \(\mathcal{L}\) 中选取 Top-\(K\) 个房源 \(\hat{\mathcal{R}}_u \subset \mathcal{L}\)，使其最大程度满足用户的显式偏好条件与隐式行为偏好。

本系统的**核心方法**为**基于内容的相似推荐（Content-Based Filtering）**：对房源多维属性做分组特征工程与标准化，经 TruncatedSVD 降维后在潜在空间计算余弦相似度，离线预计算房源间相似度矩阵 \(\mathbf{S} \in \mathbb{R}^{N \times N}\)（行为数据不足时 \(\mathbf{S}\) 即内容相似度矩阵）。在此基础上，推荐问题分解为三个子任务：

1. **基于条件的匹配推荐 + 相似度扩展**：用户通过前端界面显式选择出行目的 \(p\)、核心设施需求 \(\mathcal{F}\)、价格区间 \([P_{\min}, P_{\max}]\) 等约束条件，系统对候选房源逐一计算多维条件匹配度 \(s(l \mid p, \mathcal{F}, P)\)，并利用预计算的**内容（或混合）相似度矩阵**对高匹配「种子」房源进行邻域扩展，丰富推荐池。
2. **基于用户历史与相似度矩阵的个性化推荐**：对于已登录且存在收藏、浏览记录的用户，系统对用户曾交互过的房源，从 \(\mathbf{S}\) 中取出对应行并按行为权重加权聚合，得到候选排序（在纯内容矩阵下，等价于「在历史偏好房源的内容相似邻域内发现候选」）。
3. **热门排行推荐（冷启动兜底）**：当用户未登录、未选择条件且无可用历史，或矩阵不可用时，系统降级为基于评分与收藏热度的排行推荐。

在线推理中，策略 1 与策略 2 均依赖预计算矩阵完成扩展或聚合；**训练阶段**在交互稀疏时仅构建内容相似度，保证与论文「内容推荐为主线」的叙述一致。

## 2 系统总体架构

推荐系统采用**离线训练 + 在线推理**的经典两阶段架构：

| 阶段 | 组件 | 职责 |
|------|------|------|
| 离线训练 | `build_recommendation_model.py` | 从 MySQL 加载房源属性与用户行为数据，构建以内容相似度为基底的相似度矩阵（可选融合行为相似度），输出模型产物至 `models/` 目录 |
| 模型管理 | `ModelManager` | 服务启动时自动加载最新模型文件（`.npz` 相似度矩阵 + `.json` ID 映射），提供统一的模型访问接口 |
| 在线推理 | `RecommendationService` | 接收前端请求，根据用户条件或行为选择推荐策略，查询相似度矩阵与数据库，返回排序后的推荐列表 |
| API 层 | FastAPI `recommend.py` | 解析 HTTP 请求参数，路由至对应推荐方法，统一响应格式 |
| 前端展示 | React `Recommendation.tsx` | 提供偏好条件选择表单，调用推荐接口并渲染结果列表 |

模型产物包括：

- `listing_similarity_latest.npz`：CSR 稀疏格式的 \(N \times N\) 相似度矩阵
- `listing_id_map_latest.json`：房源 ID 与矩阵索引的双向映射（`id_to_index`、`index_to_id`）
- `recommendation_meta_latest.json`：模型元数据（方法论、参数、评估指标等）

## 3 离线模型训练——以内容相似度为核心

本章阐述推荐系统**离线相似度矩阵**的训练流程。**主干**为基于内容的过滤（Content-Based Filtering, CBF）：分组特征、SVD 降维、潜在空间余弦相似度。当用户—房源交互达到脚本内置阈值时，可**额外**计算基于隐式反馈的物品—物品相似度，并与内容矩阵加权融合（见 3.4–3.5）；交互不足时自动跳过，矩阵即为纯内容相似度。**论文若以 CBF 为主线，可将 3.2–3.3 与第 4 章评估作为核心章节，3.4–3.5 标为可选扩展或工程说明。**

### 3.1 数据来源

训练数据来源于 MySQL 业务数据库，包含两类数据源：

- **房源属性数据**：优先从 `raw_listings` 表加载（该表含已提取的设施布尔列），若不可用则回退至 `listings` 表并从 `house_tags` JSON 字段解析设施信息。筛选条件为 `price > 0`，保证数据有效性。
- **用户行为数据**：从 `favorites`（收藏表）和 `user_view_history`（浏览历史表）加载。收藏行为赋予强信号权重 \(w_{\text{fav}}=3.0\)，浏览行为赋予弱信号权重 \(w_{\text{view}}=1.0 + \min(\text{duration}/60, 2.0)\)，其中浏览时长以分钟计算，上限为 2.0 的额外加成。

### 3.2 特征工程

#### 3.2.1 特征分组与领域权重

本系统将房源特征划分为五个语义组，并为每组设定领域权重。分组策略避免了高维设施特征淹没低维但信息量大的结构/价格特征：

| 特征组 | 特征维度 | 权重 \(w_g\) | 包含特征 |
|--------|---------|-------------|---------|
| 结构特征 (structural) | 5 | 0.30 | `bedroom_count`, `bed_count`, `area`, `capacity`, `bathroom_count` |
| 价格特征 (price) | 2 | 0.15 | `price`, `price_per_bedroom`（派生特征：日价 / 卧室数） |
| 质量评价特征 (quality) | 5 | 0.15 | `rating`, `comment_count`, `favorite_count`, `heat_score`, `pic_count` |
| 设施特征 (facility) | 13 | 0.25 | 12 个设施布尔列 + `facility_ratio`（设施覆盖率） |
| 位置特征 (location) | \(d\) | 0.15 | `district` 独热编码（维度取决于商圈数量） |

其中，质量评价特征中的 `comment_count`、`favorite_count`、`heat_score` 在标准化前经过对数变换 \(\log(1+x)\)，以缓解长尾分布的影响。

#### 3.2.2 分组标准化与权重缩放

每组特征独立进行 Z-Score 标准化（`StandardScaler`），消除量纲差异。标准化后乘以 \(\sqrt{w_g}\)，使得在后续余弦相似度计算中，各组特征对最终相似度的贡献与预设权重一致。其数学依据为：

\[
\cos(\mathbf{x}, \mathbf{y}) = \sum_{g \in \mathcal{G}} w_g \cdot \cos(\mathbf{x}_g, \mathbf{y}_g)
\]

当 \(\mathbf{x}_g\) 已标准化且等长时，乘以 \(\sqrt{w_g}\) 后的内积恰好等价于加权余弦相似度。

#### 3.2.3 设施特征解析

设施特征通过关键词匹配从房源标签中提取。定义关键词映射表 `FACILITY_KEYWORDS`，对 `house_tags` JSON 中的标签文本逐一匹配，生成 12 维二值向量。额外计算设施覆盖率 `facility_ratio = Σ(设施命中) / 12` 作为补充特征，反映房源整体配套水平。

### 3.3 内容相似度计算

#### 3.3.1 TruncatedSVD 降维

将 \(N \times D\) 的特征矩阵（\(D\) 为总特征维度）输入截断奇异值分解（TruncatedSVD），提取 \(k\) 个潜在因子（默认 \(k=50\)）：

\[
\mathbf{X} \approx \mathbf{U}_k \boldsymbol{\Sigma}_k \mathbf{V}_k^\top
\]

降维的目的有二：（1）消除特征噪声，保留主要方差结构；（2）缓解高维稀疏空间中余弦相似度趋同（"维度灾难"）的问题。实际因子数取 \(\min(k, D-1, N-1)\)，保证数值稳定性。

#### 3.3.2 潜在空间余弦相似度

在 \(k\) 维潜在因子空间中计算房源间的余弦相似度：

\[
S_{\text{content}}(l_i, l_j) = \cos(\mathbf{u}_i, \mathbf{u}_j) = \frac{\mathbf{u}_i \cdot \mathbf{u}_j}{\|\mathbf{u}_i\| \cdot \|\mathbf{u}_j\|}
\]

其中 \(\mathbf{u}_i = \mathbf{U}_k[i, :]\) 为第 \(i\) 套房源的潜在因子表示。

### 3.4 可选：行为相似度（隐式反馈，类 Item-Item 方法）

本节描述训练脚本中的**可选**分支：与 3.3 节的内容相似度（仅依赖房源属性）不同，行为相似度基于**用户群体的隐式反馈**——「与同一用户发生过交互的房源」在向量空间上更接近。实现上采用 User–Item 矩阵经 IUF 加权后的物品余弦相似度，与经典 Item-based 邻域方法一致。**若论文不展开协同过滤，可将本节作为「数据充足时的扩展」或附录，不纳入创新点表述。**

#### 3.4.1 User-Item 隐式反馈矩阵

从收藏与浏览记录构建 \(M \times N\) 的隐式反馈矩阵 \(\mathbf{R}\)（\(M\) 为用户数），元素 \(r_{u,i}\) 为用户 \(u\) 对房源 \(l_i\) 的交互权重（收藏 3.0，浏览 1.0 + 时长奖励）。与显式评分矩阵不同，隐式反馈矩阵的元素为置信度权重而非评分值，更适合本系统"无评分、有行为"的数据特点。

#### 3.4.2 IUF 加权

直接基于 \(\mathbf{R}\) 计算的物品相似度会受到热门房源偏差的影响——被大量用户交互的热门房源会与其他房源产生虚假的高相似度。为此引入逆用户频率（Inverse User Frequency, IUF）加权：

\[
\text{IUF}_j = \log \frac{M}{|\{u : r_{u,j} > 0\}| + 1}
\]

对 \(\mathbf{R}\) 的每一列乘以对应的 IUF 权重，使热门物品的影响被对数压缩。

#### 3.4.3 Item-Item 余弦相似度

在 IUF 加权后的矩阵上转置，计算物品间的余弦相似度：

\[
S_{\text{behavior}}(l_i, l_j) = \cos(\mathbf{r}'_i, \mathbf{r}'_j)
\]

其中 \(\mathbf{r}'_i\) 为 IUF 加权后的物品向量。当行为数据不足（用户数 < 3 或交互数 < 10）时，跳过行为相似度计算。

### 3.5 可选：内容矩阵与行为矩阵的加权融合

将内容相似度 \(\mathbf{S}_{\text{content}}\) 与行为相似度 \(\mathbf{S}_{\text{behavior}}\) 通过加权混合（Weighted Hybridization）融合（Burke, 2002 中的线性加权类别）：

\[
\mathbf{S}_{\text{hybrid}} = \alpha \cdot \mathbf{S}_{\text{content}} + (1 - \alpha) \cdot \mathbf{S}_{\text{behavior}}
\]

默认融合系数 \(\alpha = 0.7\)，即内容相似度占主导。融合后进行 Min-Max 归一化至 \([0, 1]\) 区间，并将对角线元素置为 1.0。

当行为数据不足时，脚本跳过 \(\mathbf{S}_{\text{behavior}}\)，矩阵即为 \(\mathbf{S}_{\text{content}}\)，保证推荐结果始终可用。该设计是**冷启动与稀疏数据下的工程策略**：论文主线写 CBF 时，可强调「行为融合为可选增强，非本文实验必要条件」。

### 3.6 模型参数汇总

| 参数 | 符号 | 默认值 | 说明 |
|------|------|--------|------|
| 融合系数 | \(\alpha\) | 0.7 | 内容相似度权重；行为权重为 \(1-\alpha\) |
| SVD 因子数 | \(k\) | 50 | TruncatedSVD 的潜在因子维度 |
| 评估 Top-K | \(K\) | 20 | 离线评估时取前 K 个推荐结果 |
| 收藏权重 | \(w_{\text{fav}}\) | 3.0 | 收藏行为在隐式反馈矩阵中的权重 |
| 浏览权重 | \(w_{\text{view}}\) | 1.0 + 时长奖励 | 浏览行为权重，时长奖励上限 2.0 |
| 行为数据阈值 | — | ≥ 3 用户, ≥ 10 交互 | 低于此阈值则跳过行为相似度 |

## 4 离线评估

### 4.1 内容一致性评估

内容一致性评估不依赖用户行为数据，直接基于相似度矩阵与房源属性进行。对每套房源取其 Top-\(K\) 相似房源，计算以下指标：

| 指标 | 定义 | 含义 |
|------|------|------|
| Coverage@K | \(\frac{|\bigcup_i \text{Top}_K(l_i)|}{N}\) | 推荐覆盖率，衡量模型是否将推荐集中在少数热门房源 |
| Avg\_Similarity@K | \(\frac{1}{N}\sum_i \overline{S(l_i, \text{Top}_K(l_i))}\) | 平均相似度，过高表示区分度不足，过低表示推荐相关性差 |
| District\_Consistency | Top-K 中与基准房源同商圈的比例 | 地理位置一致性，反映位置特征的有效性 |
| Price\_MAE\_Ratio | \(\frac{1}{N}\sum_i \frac{|\text{mean}(P_{\text{Top}_K}) - P_i|}{P_i}\) | 价格相对偏差，衡量推荐房源与基准房源的价格匹配程度 |
| Diversity@K | \(1 - \overline{S_{\text{pairwise}}(\text{Top}_K)}\) | Top-K 内部多样性，越高表示推荐结果越丰富 |

### 4.2 留一法（Leave-One-Out）评估

针对收藏数 ≥ 2 的用户执行留一法交叉验证：依次隐藏用户的一条收藏，以剩余收藏房源的相似度向量均值作为推荐依据，检测被隐藏的房源是否出现在 Top-\(K\) 推荐列表中。

| 指标 | 定义 | 含义 |
|------|------|------|
| HitRate@K | 命中次数 / 总测试次数 | 被隐藏的收藏房源出现在 Top-K 中的概率 |
| MRR | \(\frac{1}{T}\sum_t \frac{1}{\text{rank}_t}\) | 平均倒数排名，衡量命中位置的靠前程度 |

复现时，实际评估结果以 `models/recommendation_meta_latest.json` 中的 `evaluation` 字段为准。

## 5 在线推理服务

### 5.1 推荐策略分层

在线推理服务（`RecommendationService`）内部实现了三种策略；**HTTP 层实际路由顺序**以 [`app/api/endpoints/recommend.py`](../app/api/endpoints/recommend.py) 为准（与下述流程图略有不同）：

**真实做法（API 入口）**：`GET /api/recommend` **先判断**「合并登录用户补全后的请求」是否含 **`travel_purpose` 或设施列表」。若 **有**，则 **只** 调用策略 1（`get_condition_based_recommendations`），**不会**再进入 `get_recommendations` 中的策略 2。仅当 **两者皆无** 时，才调用 `get_recommendations`，在其中依次尝试策略 2 → 策略 3。

因此：**前端「个性化推荐」页默认带有出行目的（如 `couple`）时，线上主路径是策略 1，而非基于收藏/浏览的协同过滤（策略 2）。** 详细说明与答辩口径见 [`RECOMMENDATION_ONLINE_BEHAVIOR.md`](./RECOMMENDATION_ONLINE_BEHAVIOR.md)。

下列流程图描述的是 **服务内部能力分层**（便于理解 1/2/3 的差异），**不等价于**「每次请求按 2、3 顺序尝试」：

```
请求到达（经 API 路由后）
  │
  ├── 已判定走条件分支？（travel_purpose 或 facilities 非空）
  │     └── 是 → 策略 1：基于条件的智能推荐
  │
  ├── 否则进入 get_recommendations：
  │     ├── 用户已登录且有行为数据且矩阵可用？
  │     │     └── 是 → 策略 2：基于用户历史与相似度矩阵的个性化推荐
  │     └── 否则 → 策略 3：热门排行推荐（冷启动兜底）
```

### 5.2 策略 1：基于条件的智能推荐

该策略面向用户显式选择偏好条件的场景（如前端"个性化推荐"页面），执行流程如下：

**Step 1：条件融合**

将用户显式选择的设施需求与出行目的隐含的设施偏好合并。出行目的与设施偏好的映射关系如下：

| 出行目的 | 隐含设施偏好 | 价格倾向系数 |
|---------|------------|------------|
| 情侣出游 (couple) | 投影、浴缸、阳台 | 1.0 |
| 家庭亲子 (family) | 厨房、洗衣机、WiFi | 1.0 |
| 商务差旅 (business) | WiFi、近地铁、智能锁 | 1.2（偏好高端） |
| 学生考研 (exam) | WiFi、近地铁 | 0.6（偏好经济） |

**Step 2：候选集筛选**

从数据库中按硬约束条件（价格区间、商圈、卧室数）查询候选房源，上限 500 条，按评分降序排列。

**Step 3：多维匹配度评分**

对每个候选房源计算综合匹配分数：

\[
s(l) = 0.40 \times s_{\text{facility}} + 0.25 \times s_{\text{purpose}} + 0.20 \times s_{\text{quality}} + 0.15 \times s_{\text{price}}
\]

各分项的计算方式：

- **设施匹配分 \(s_{\text{facility}}\)**：用户明确要求的设施中被房源命中的比例。若用户未指定设施则默认 0.5。
- **目的匹配分 \(s_{\text{purpose}}\)**：出行目的隐含的设施偏好中被命中的比例。若无出行目的则默认 0.5。
- **质量分 \(s_{\text{quality}}\)**：\(0.7 \times \frac{\text{rating}}{5.0} + 0.3 \times \min\left(\frac{\log(1+\text{fav\_count})}{7}, 1\right)\)
- **价格适配分 \(s_{\text{price}}\)**：价格越接近用户偏好区间的中心点越高，\(1 - \frac{|P_l - P_{\text{mid}}|}{P_{\text{range}}}\)，其中中心点受出行目的的价格倾向系数调节。

**Step 4：相似度扩展**

取匹配分最高的 3 套房源作为"种子"，从预计算的相似度矩阵中提取其 Top-7 相似房源。相似扩展房源的得分为：

\[
s_{\text{expanded}} = s_{\text{seed}} \times S(l_{\text{seed}}, l_{\text{new}}) \times 0.85
\]

乘以衰减因子 0.85 以区分直接匹配与相似度扩展的推荐来源。扩展结果需通过硬约束过滤。

**Step 5：合并排序**

将直接匹配的 Top-K 结果与相似度扩展结果合并去重，按综合分数降序取 Top-K 返回。

### 5.3 策略 2：基于用户历史与相似度矩阵的个性化推荐

面向已登录用户且存在收藏或浏览记录的场景，系统利用离线预计算的相似度矩阵 \(\mathbf{S}\)（稀疏数据下即为内容相似度矩阵），通过以下流程完成个性化推荐：

1. **历史召回**：查询用户的收藏记录（权重 3.0）和最近 20 条浏览记录（权重 1.0），构建交互历史集合 \(\mathcal{I}_u = \{(l_j, w_j)\}\)。
2. **矩阵聚合评分**：对每条交互记录，从 \(\mathbf{S}\) 中取出对应行的相似度向量，按行为权重加权平均，得到候选房源分数：
   \[
   \mathbf{score} = \frac{\sum_{(l_j, w_j) \in \mathcal{I}_u} w_j \cdot \mathbf{S}[j, :]}{\sum w_j}
   \]
   在**纯内容矩阵**下，可表述为：在用户已偏好的房源周围，按内容相似度扩散候选；若训练阶段已融合行为相似度，则矩阵同时携带群体共现信息。**论文若以 CBF 为主线，宜以前一句为主表述，避免将本策略单独命名为「协同过滤章」。**
3. **去重过滤**：排除用户已交互过的房源，避免重复推荐。
4. **排序返回**：按聚合分数降序排列，应用商圈、价格等可选筛选条件后取 Top-K。

若用户行为数据为空或所有交互房源均不在模型中，则该策略返回空结果，自动降级至策略 3。

### 5.4 策略 3：热门排行推荐（冷启动兜底）

当策略 1 和策略 2 均不适用时（即用户未选择条件、未登录或无行为数据），系统执行热门排行推荐：

\[
s_{\text{hot}}(l) = 0.7 \times \frac{\text{rating}}{5.0} + 0.3 \times \min\left(\frac{\text{fav\_count}}{100}, 1\right)
\]

按评分降序从数据库查询候选房源，计算热度分后排序返回。该策略不依赖任何用户信息或预训练模型，保证在任何情况下均能返回有效推荐。

### 5.5 相似房源推荐

除主推荐接口外，系统还提供独立的"相似房源"接口（`/api/recommend/similar/{homestay_id}`）：

1. 优先使用预计算的相似度矩阵进行 O(1) 查询。
2. 若目标房源不在矩阵中（如新上架的房源），回退至 SQL 查询：按同商圈 + 价格浮动 ±30% + 评分降序的规则查找相似房源。

## 6 冷启动策略

冷启动（Cold Start）是推荐系统面临的经典挑战，指系统缺乏足够的用户行为数据来进行个性化推荐的情形。本系统在多个层面设计了冷启动应对策略：

### 6.1 当前数据现状与冷启动分析

作为毕业设计项目，平台尚处于开发与演示阶段，用户行为数据积累有限。截至本文撰写时，数据库中的用户交互数据如下：

| 数据表 | 记录数 | 用户数 | 涉及房源数 |
|--------|--------|--------|-----------|
| `favorites`（收藏） | 8 | 7 | 2 |
| `user_view_history`（浏览） | 0 | 0 | — |

训练脚本设定了行为数据的最低阈值：需至少 3 位用户且至少 10 条交互记录，方可计算行为相似度并参与融合。当前数据（7 用户、8 条交互）未达到交互数阈值，因此**在实际训练中，行为相似度模块被跳过，矩阵完全基于内容特征构建**。

这一现象正是**冷启动与稀疏交互**下的常态：平台初期难以依赖共现信号。本系统通过**自动降级**保证矩阵与在线服务始终可用，与论文「以内容相似度与条件匹配为核心」的叙述一致。

### 6.2 模型层面的冷启动机制

在离线训练阶段，系统通过以下逻辑实现自适应降级：

```
if 用户数 < 3 or 交互数 < 10:
    S_hybrid = S_content          # 纯内容相似度 (α=1.0)
else:
    S_hybrid = α × S_content + (1−α) × S_behavior  # 混合融合
```

这意味着：
- **即使没有任何用户行为数据**，系统仍然能够基于房源属性（结构、价格、质量、设施、位置）提供合理的相似度推荐。
- **当行为数据逐渐积累**并超过阈值后，重新训练即可自动启用 3.4–3.5 中的行为分支与融合，无需修改业务代码。

从系统演化角度看，这是**渐进式策略**：默认以内容相似度为主；数据充足时再引入行为相似度参与矩阵构建（论文可选写为展望或附录实验）。

### 6.3 服务层面的冷启动

推荐策略的三级降级机制本身即构成完整的冷启动链：

| 场景 | 触发条件 | 降级策略 |
|------|---------|---------|
| 新用户 + 显式条件 | 用户选择了出行目的或设施需求 | 策略 1（条件匹配 + 矩阵扩展），无需行为数据 |
| 新用户 + 无条件 | 用户未登录或未选择任何条件 | 策略 3（热门排行），基于全局统计 |
| 老用户 + 行为不在模型中 | 用户有行为，但交互的房源不在矩阵中 | 策略 3（热门排行），等待模型重训练纳入新数据 |
| 老用户 + 历史可用 | 用户有收藏/浏览记录且房源在矩阵中 | 策略 2（历史 + 矩阵聚合） |

### 6.4 数据增长与推荐演化

模型不会自动重新训练。若希望矩阵包含行为相似度分量，需在交互数据超过阈值后手动执行训练脚本：

```bash
python scripts/build_recommendation_model.py --alpha 0.7 --n-factors 50 --top-k 20
```

推荐系统的演化路径如下：

| 阶段 | 数据条件 | 相似度矩阵组成 | 在线推荐策略 |
|------|---------|---------------|------------|
| 冷启动期（当前） | 交互 < 10 | 100% 内容相似度 | 以条件匹配（策略 1）和热门排行（策略 3）为主 |
| 过渡期 | 10 ≤ 交互 < 100 | 70% 内容 + 30% 行为 | 矩阵含行为分量；策略 2 的邻域信息更丰富 |
| 成熟期 | 交互 ≥ 100，可降低 α | 30% 内容 + 70% 行为 | 行为分量在矩阵中权重提高（可选调参） |

需要指出的是，即使在相似度矩阵不更新的情况下，**推荐结果仍然是动态变化的**：

- 策略 1 实时查询数据库中的最新房源与标签数据。
- 策略 2 的用户行为查询是实时的——用户新增的收藏和浏览会立即改变聚合所用的历史与权重。
- 策略 3 依据实时评分与收藏量排序。

因此，推荐结果能够随用户行为与房源数据更新而动态调整；模型重训练则用于更新矩阵中行为分量的有无与强弱。

## 7 API 接口设计

### 7.1 推荐接口

| 方法 | 路径 | 功能 |
|------|------|------|
| `GET` | `/api/recommend/` | 主推荐接口（条件推荐 / 历史矩阵聚合 / 热门排行自动路由） |
| `POST` | `/api/recommend/` | 主推荐接口（POST 方式） |
| `GET` | `/api/recommend/similar/{homestay_id}` | 基于房源 ID 的相似房源推荐 |
| `GET` | `/api/recommend/popular` | 热门房源排行 |
| `GET` | `/api/recommend/personalized` | 基于用户偏好设置的个性化推荐 |

### 7.2 主推荐接口参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `travel_purpose` | `string` | 出行目的：`couple` / `family` / `business` / `exam` / `team_party` / `medical` / `pet_friendly` / `long_stay` |
| `facilities` | `string` | 核心设施需求，逗号分隔：`subway,projector,bathtub,cooking` 等 |
| `district` | `string` | 商圈筛选 |
| `price_min` / `price_max` | `float` | 价格区间 |
| `bedroom_count` | `int` | 最少卧室数 |
| `top_k` | `int` | 返回数量（1-50，默认 10） |

当请求携带 `travel_purpose` 或 `facilities` 参数时，触发条件匹配推荐（策略 1）；否则根据用户登录状态选择策略 2（有历史则矩阵聚合）或策略 3（热门兜底）。

**场景标签与医院距离融合（可选）**：若 MySQL 中房源已写入 **`scene_scores`**（离线 `scripts/listing_scene_pipeline.py`：弱监督含文本 + 医院 POI 地理信号，TF-IDF + 多标签逻辑回归，详见 [`LISTING_SCENE_TFIDF_PIPELINE.md`](LISTING_SCENE_TFIDF_PIPELINE.md)），策略 1 在综合匹配分上增加 **`0.03 × P(travel_purpose | 房源)`**（概率截断于 \([0,1]\)）。另若同表已写入 **`nearest_hospital_km`**（流水线按 POI 与经纬度离线计算），且当前请求 **`travel_purpose=medical`**，再叠加 **距离加分**（≤2km 满幅、2–5km 线性衰减至 0，具体常数见 `recommender.py` 与场景文档）。列为空或未跑流水线时对应项为 0。在线服务**不**在请求路径加载 `joblib`、**不**实时重算医院距离。推荐与列表接口可将 `nearest_hospital_km` 返回前端作展示。

### 7.3 响应格式

```json
{
  "recommendations": [
    {
      "id": "100230812",
      "unit_id": "100230812",
      "title": "江景复式大平层",
      "district": "武昌区",
      "price": 328.0,
      "rating": 4.9,
      "cover_image": "https://...",
      "facilities": ["投影", "浴缸", "可做饭"],
      "match_score": 0.87,
      "reason": "适合情侣出游 | 有投影、有浴缸"
    }
  ]
}
```

其中 `match_score` 为归一化至 \([0, 1]\) 的综合推荐分数，`reason` 为自然语言推荐理由，便于前端展示。

## 10 算法理论依据

**论文若以基于内容的推荐为主线**，可优先引用以下文献支撑特征降维与相似度构造：

1. **潜在因子与矩阵分解思想**：Koren Y., Bell R., Volinsky C. "Matrix Factorization Techniques for Recommender Systems", *IEEE Computer*, 2009. 本系统借鉴其「低维潜在空间」思路，采用 TruncatedSVD 对房源特征矩阵降维后计算余弦相似度。

2. **基于内容的推荐**：经典表述见 Ricci et al. 等编 *Recommender Systems Handbook* 中 Content-Based 章节。本系统通过分组特征、领域权重与标准化，将房源属性映射为可比的向量表示。

**若正文需要交代训练脚本中的可选行为分支**，可补充：

3. **物品邻域与隐式反馈**：Sarwar B. et al. "Item-Based Collaborative Filtering Recommendation Algorithms", *WWW*, 2001 — 行为相似度在实现上属于 Item–Item 邻域类方法。

4. **加权混合框架**：Burke R. "Hybrid Recommender Systems: Survey and Experiments", *UMUAI*, 2002 — 用于说明 3.5 节线性融合在推荐系统分类中的位置（可不作为本文创新点）。

## 11 方案选型论证

### 11.1 为什么选择「内容相似度 + 条件匹配」作为主线

**（1）冷启动与数据规模现实**

民宿毕设/演示环境中，用户—房源交互往往稀疏，难以支撑以共现为核心的纯协同方法。以**房源属性**为输入的内容相似度不依赖大规模日志，可在全生命周期内稳定产出矩阵与推荐。

**（2）业务可解释性**

出行目的、设施、价格区间等条件与房源标签、结构化字段直接对应，便于在论文中说明**匹配函数设计**与**相似扩展**的合理性，满足经管或工科类毕设对「需求—实现」可追溯的要求。

**（3）工程与复杂度匹配**

离线预计算 \(N \times N\) 相似度矩阵（稀疏存储），在线以行查询 + Top-K 为主，复杂度可控；相较端到端深度学习，更适合毕设时间与算力。

**（4）与代码实现一致**

当前默认训练路径在交互不足时即**不启用**行为相似度分支，论文主线写 CBF + 条件匹配，与 `recommendation_meta_latest.json` 及线上行为无需强行对齐「协同过滤」叙事。

### 11.2 各技术选型的学术依据（主干）

| 技术选型 | 学术出处 / 传统类别 | 选用理由 |
|---------|-------------------|---------|
| 分组特征 + 领域权重 + 标准化 | 特征工程通法；可对照 Handbook 中 CBF 表述 | 避免单一维度淹没多维属性，匹配民宿「结构/价格/质量/设施/位置」语义 |
| TruncatedSVD 降维 | Koren et al., *IEEE Computer*, 2009 等潜在因子工作 | 缓解高维下相似度区分度不足的问题 |
| 条件匹配 + 矩阵扩展 | 规则/学习混合的常见产品形态 | 显式需求由匹配分表达，扩展由 \(\mathbf{S}\) 提供多样性 |
| 留一法（Leave-One-Out） | Herlocker et al., *ACM TOIS*, 2004 等 | 交互较少时最大化利用收藏等行为做离线验证（可选报告） |

行为分支中的 **IUF**、**Item–Item 余弦** 可在附录中引用 Sarwar（2001）与 TF–IDF 类文献。

### 11.3 与替代方案的对比

| 方案 | 优势 | 劣势 | 与本项目关系 |
|------|------|------|-------------|
| **纯协同 / 邻域 CF** | 能反映共现偏好 | 稀疏即失效 | 作可选矩阵分量，非本文主线 |
| **纯内容（CBF）** | 无冷启动、可解释 | 难利用共现（数据多时） | **本文主干** |
| **矩阵分解（ALS 等）** | 用户—物品联合建模 | 需较多交互、调参与工程量大 | 可作展望 |
| **深度学习推荐** | 表达能力强 | 数据与算力要求高 | 可作展望 |

## 12 结论性表述建议

可在论文小结中强调（与「不提协同过滤为主线」一致）：

1. **内容相似度的有效性**：通过分组特征、SVD 降维与余弦相似度，构建房源间可解释的相似关系，支撑相似房源与扩展推荐。
2. **条件匹配与相似扩展的结合**：显式条件决定候选与种子排序，矩阵扩展在约束内丰富结果，兼顾可控性与多样性。
3. **多级服务策略**：条件推荐、历史矩阵聚合、热门兜底形成完整链路，适应未登录、无历史、矩阵缺失等情况。
4. **稀疏数据下的训练策略**：交互不足时自动仅训练内容矩阵，与论文问题背景一致，避免夸大协同信号。
5. **工程可行性**：离线预计算、在线行查询 + Top-K，满足 Web 响应需求。

若需保留一句「扩展性」：**训练脚本已实现行为相似度的可选融合，待数据积累后可作为后续工作与对比实验**，而不必上升为本文核心贡献。
