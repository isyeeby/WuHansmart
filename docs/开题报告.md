一、 课题研究的背景及意义
1.1 研究背景
近年来，我国数字经济蓬勃发展，“互联网 + 旅游” 模式深度融合，共享住宿（民宿）作为新兴非标准住宿业态，已成为推动旅游消费升级的重要力量。据国家信息中心分享经济研究中心《中国共享住宿发展报告》显示，2023 年我国共享住宿市场交易规模突破数百亿元，乡村民宿和城市特色民宿订单量较疫情前实现倍数级增长，行业正从 “野蛮生长” 迈向 “品质化、数字化、精细化” 运营新阶段。
武汉作为 “九省通衢” 的中部中心城市，文旅资源丰富，坐拥大量高校与在校大学生群体，商旅与休闲客流并存，造就了结构多元的民宿市场：既有以高校周边为代表的刚性租住需求，也有以核心商圈与景区为代表的节假日脉冲式需求，价格弹性与空间分异显著。在数据获取层面，主流民宿平台安全策略趋严，全网大规模、高频率采集难度大，而限定城市与单一高价值平台、结合浏览器自动化与接口监听的深度采集，有助于在合规前提下获得可分析样本，使武汉成为具有地域代表性且工程上可实施的实验场景。
然而，行业高速发展背后，个体民宿经营者普遍面临“数据焦虑”与“决策盲区”。主流 OTA 平台掌握交易数据但对商家端开放有限，房东往往只能获取挂牌价与少量公开信息，缺乏区域成交均价、市场热度、竞品趋势等关键指标支撑，导致旺季定价保守、淡季空置率上升等决策困难。在成交数据不可得的现实约束下，探索基于公开挂牌价、评价与收藏等可观测信号的市场热度刻画、商圈价格规律分析与机器学习辅助定价方法，并将其系统化落地为可交互平台，具有明确的研究价值与应用意义。

1.2 研究意义
1.2.1 理论意义
本课题从数据受限场景出发，丰富民宿价格分析与数据驱动运营研究。首先，在平台成交数据不可得的条件下，引入代理变量建模思路，探索使用可观测信号估计市场热度与需求变化，为相关研究提供可行的数据补全路径。其次，聚焦武汉“高校密集 + 交通枢纽”双属性城市，分析商圈间价格空间异质性与典型时段波动规律，补充对该类城市市场结构的研究。最后，将离线统计分析与机器学习预测方法结合，构建面向经营决策的闭环分析框架，提升研究的可解释性与落地性。
1.2.2 实践意义
本课题的研究具有显著的实践价值，能够为民宿经营者、投资者、监管部门及平台方带来多方面积极影响。对于民宿经营者而言，系统提供的智能定价预测、竞品与商圈对比、房源列表与详情浏览、收藏与个人问卷偏好、我的房源维护与定价建议等功能，可帮助其降低信息搜寻成本，理解区域行情与设施溢价，避免旺季利润流失与淡季房源空置。对于潜在投资者，系统输出的行政区与商圈层面的统计看板、价格机会（相对模型估计偏低）辅助展示、投资沙盘与敏感性分析等，可为选址与收益预期提供数据参考，降低盲目投资风险。对于监管部门，系统的动态化、可视化监控能力有助于从宏观上把握民宿供给与价格分布态势，为行业监测与政策研究提供辅助信息。对于民宿平台方，公开数据层面的分析工具可协助理解区域市场结构，与平台自身数据体系形成互补认知。总体而言，本课题强调在合规采集与最小必要存储前提下完成系统实现，具体功能与当前开发项目保持一致。

二、 研究现状及发展趋势
2.1 研究现状	
2.1.1 民宿价格影响因素与空间分异研究
国内外学者已证实民宿价格受多维度因素影响，且存在显著空间异质性。多尺度地理加权回归（MGWR）模型[2]实证表明，共享住宿价格空间分异特征与房源自身功能属性、区域发展定位、消费人群结构等商圈核心属性相关，且调控作用在不同城市类型中存在差异；房源面积、卧室数量等硬件特征对价格的影响系数在微观尺度上更稳定，交通可达性、周边配套等外部因素的影响权重随商圈类型动态变化，这一特征在一线城市尤为突出。针对生态涵养区的研究发现，民宿价格空间分异受房源特征、建筑特征、自然环境及区位特征综合影响，高程、山景可视度等自然要素的解释力显著高于传统区位因子，且在山地与平原区域作用强度相反[1]。全球多城市对比研究则证实，民宿价格形成机制具有空间依赖性，房源功能特征、区位条件与区域消费能力三类核心因素的影响权重在不同城市、同一城市不同商圈间存在显著差异，导致单一定价模型难以适配多区域场景。
现有成果多集中于北京、上海等一线城市及国外市场，对武汉这类 “高校密集 + 交通枢纽” 双属性城市关注不足，商圈层面的精细化价格机制分析存在明显研究缺口。
2.1.2 民宿价格预测与数据处理技术
在价格预测算法方面，XGBoost 算法凭借对高维非线性数据的强拟合能力，在民宿价格预测任务中的平均绝对误差（MAE）较传统线性回归模型降低 28.6%，且在特征稀疏场景下仍能保持稳定性能。研究还发现，民宿价格具有 “局部集聚性” 特征，同一商圈内房源的价格影响因素权重相似度较高，通过聚类预处理可使模型训练效率提升 30% 以上[3]。在数据处理与采集方面，人工智能模型能处理 “碎片化、非结构化” 多源数据，但数据时效性直接决定模型落地效果，超过 7 天未更新的数据会使预测误差扩大 40%[6]；在平台反爬机制日趋严苛的背景下，本课题定价采用 XGBoost 回归，推荐采用基于内容的相似度、条件匹配与浏览收藏等轻量行为信号。
2.1.3 大数据分析架构与Playwright浏览器自动化在研究中的应用
现有民宿大数据研究多采用“Python爬虫 + Hadoop/Hive + 可视化”架构。分层存储与离线聚合能够同时满足数据持续更新与多维统计分析需求；联机业务常依赖关系型数据库支撑用户、房源列表与鉴权等低延迟访问。在页面强交互、接口返回结构化的移动站点场景中，基于 Playwright 等工具的浏览器自动化与网络响应监听，可在合规频控前提下稳定获取列表与详情中的价格日历、标签等数据，较单纯静态解析更具适应性。推荐领域，基于内容的过滤通过物品特征相似度与显式条件约束相结合，在用户—物品交互稀疏时仍可提供可解释结果；本课题实现上以内容相似度矩阵、截断奇异值分解与余弦相似度、条件打分与可选行为加权为主线，首页与个性化推荐页在接口层分治，以贴合当前产品形态与数据条件。

2.2 发展趋势
未来民宿价格数据分析将呈现“限定范围可持续采集 + 业务库与离线数仓并存 + 离线训练与在线推理分离”的技术组合趋势；模型方面将更多采用集成学习并强调分区域、分场景的适配性与可解释性；推荐系统将更多结合内容特征、问卷与浏览收藏等轻量行为信号，并与条件匹配策略相配合；应用方面将向闭环化与场景化发展，通过驾驶舱、热力图、趋势与对比看板降低使用门槛，同时更重视数据安全、合规采集与隐私最小化原则。
三、 课题研究目标及内容
3.1 研究目标
本课题旨在利用 Hive  离线分析能力、Python 浏览器自动化采集与机器学习算法，构建基于公开数据的民宿价格分析与支持系统。具体目标包括：一是形成可维护、可断点续传的途家武汉区域采集程序，获取房源列表摘要、价格日历与详情标签等数据，并支持增量持久化与备份；二是完成数据清洗、入库与 Hive 分层导入，建立分析类服务在 Hive 与业务库之间的可配置查询与回退机制，保障统计口径一致与演示可用性；三是揭示行政区与商圈维度的价格分布、设施溢价与热度相关规律，支撑经营驾驶舱与论文描述性分析；四是训练并部署 XGBoost 价格预测模型，服务智能定价、价格机会识别、投资沙盘等环节；五是实现基于内容的推荐与条件匹配、首页智能推荐展示、登录用户下列表偏好规则重排及收藏与浏览历史等联机能力；六是完成前后端分离的 Web 系统开发与开题所涉研究内容阐述，为毕业论文撰写提供与系统实现一致的论述基础。
3.2 研究内容
3.2.1 浏览器自动化采集与流式抓取
针对途家网（Tujia）武汉区域，采用 Python 与 Playwright 启动浏览器会话，在列表页滚动加载过程中监听搜索类接口返回，对 JSON 响应进行解析，抽取房源标识、标题、行政区、地址、经纬度、挂牌价区间、评分与评论量、收藏量、封面与列表标签等摘要信息，并维护待处理队列与已见集合，避免重复抓取。对于尚未获得价格日历的房源，程序在可见区域内定位详情链接，滚动至视区后模拟点击进入详情页；在详情阶段监听价格日历接口与房源详情接口的响应，捕获日历结构与结构化详情标签，用于后续入库与特征解析。价格日历需通过页面上的日历入口触发弹层，程序在点击与等待策略上引入随机时延，并在成功后关闭弹层并返回列表页，形成“列表发现—详情拉取—返回继续滚动”的流式闭环。已处理房源集合与日历、标签数据持久化至本地 JSON。
3.2.2 数据治理、入库与基于 Hive 的离线分析
对采集获得的 JSON 与合并结果进行去重、缺失与异常规则处理、格式标准化与标签文本规范化，使业务库与后续分析口径统一。通过后端提供的导入程序将房源主数据、价格日历与详情标签等写入联机业务库，并可按项目脚本向 Hive 的 ODS、DWD、DWS、ADS 等分层表导入，形成离线主题汇总能力。分析类 HTTP 接口在部署上可配置优先使用 Hive 聚合结果，在 Hive 未启用或不可用时回退至业务库同口径 SQL 聚合，保证实验与演示环境稳定。在此基础上，围绕全市与行政区、商圈、价格区间与设施等维度开展统计，输出驾驶舱关键指标、热力分布、排行榜与商圈对比等结果。公开数据中评论量、收藏量等可作为热度或需求的代理观测，用于看板展示与横比，不强行构造不可验证的成交序列。

3.2.3 特征构建、文本场景分类与 XGBoost 价格建模
面向定价与推荐，需要从结构化字段与标签文本中构造一致的特征体系。对设施等语义可采用关键词规则将标签文本映射为二值特征，与导入阶段解析结果对齐；对标题、标签与摘要等可构造弱标签训练样本，采用 TF-IDF 与逻辑回归完成出行场景等类别的文本分类，输出分数可作为推荐排序或场景加分的辅助信号。在监督学习定价方面，采用 XGBoost 回归，结合交叉验证与超参数调整等常规手段，以挂牌价等为监督目标训练模型，导出持久化模型文件，由后端服务加载并提供推理接口，用于智能定价页面、我的房源调价建议、价格机会候选筛选以及投资测算中的价格输入等环节。模型评价可采用平均绝对误差、均方根误差、平均绝对百分比误差等回归指标，具体数值以当次训练样本与划分为准。
3.2.4 基于内容的推荐、条件匹配与联机个性化
离线阶段对房源多维特征进行分组标准化与领域权重缩放，经截断奇异值分解在潜在空间计算余弦相似度，预计算稀疏房源相似度矩阵及标识映射文件，供服务启动加载。在线推荐以用户显式条件（出行目的、设施、价格与区划等）过滤与多维匹配打分为核心路径；在相似度矩阵可用时，对高匹配种子房源进行邻域扩展以丰富结果；对已登录且存在浏览或收藏记录的用户，在请求未强约束条件或策略分支允许时，可基于历史交互在相似度矩阵上行加权聚合，生成个性化候选。首页智能推荐条与个性化推荐页在后端分属不同接口：前者以 SQL 候选集结合场景与设施等重排为主，后者以条件推荐与矩阵策略为主，二者与房源详情“相似房源”等接口在职责上相互区分。房源列表在登录状态下支持按收藏与浏览推断偏好行政区与商圈的规则化重排，该路径不读取推荐相似度矩阵，与推荐模块解耦。冷启动、未登录或矩阵未部署时，系统降级为热度、评分与收藏等兜底排序，保证服务可用。该设计遵循基于内容的过滤思路，并与显式条件匹配、行为信号增强相结合。

3.2.5 民宿价格数据分析系统实现
开发覆盖“采集—导入—离线分析—模型训练—在线服务—可视化”的链路，采用分层思路：数据层为采集持久化与入库、数仓导入；处理层为清洗、特征与弱标签流水线；模型层为 XGBoost 与推荐矩阵及可选场景分类器；应用层为 FastAPI 接口与 React、TypeScript 前端。后端提供用户鉴权、问卷与个人信息、房源列表与详情、关键词检索、个性化排序、浏览历史、收藏、首页与推荐接口、驾驶舱与分析、智能定价预测、竞品参照、我的房源、投资沙盘、价格机会、多房源对比等能力；前端使用 Ant Design 与 ECharts 等实现经营驾驶舱、商圈分析、智能定价、个性化推荐、房源列表与详情、我的房源、竞品情报、投资沙盘、价格机会、房源对比、收藏与个人中心等页面的交互与图表展示

四、 研究方案
4.1 总体设计思路
本课题遵循“数据采集与持久化—数据治理与入库—Hive 离线聚合与指标固化—模型离线训练与在线推理—Web 交互展示”的总体思路，围绕公开数据条件下热度刻画、价格预测与房源推荐展开。通过可复现脚本与统一口径的指标体系，结合离线统计与机器学习，形成可交互、可解释、可对照代码实现的民宿价格分析系统。
4.2 技术路线
数据层：采用 Playwright 浏览器自动化访问途家移动端列表与详情，监听接口响应并结合页面操作获取价格日历与详情标签，流式滚动与断点续传，JSON 分文件持久化与备份。 
数据处理层：使用 Python 完成缺失、重复与异常处理，关键词规则与 TF-IDF、逻辑回归完成设施与场景相关特征；通过导入脚本写入业务库并向 Hive 分层表导数。
模型层：XGBoost 价格回归；基于内容的相似度矩阵构建与在线多策略推荐；交叉验证与误差指标评估定价模型。  
应用层：FastAPI 提供数据查询、预测与推荐等接口；React 与 TypeScript 结合 Ant Design、ECharts 实现可视化与业务页面；JWT 与问卷、收藏、浏览历史支撑个性化能力。
4.3 功能架构图
本系统围绕“采集—入库—离线分析—模型服务—可视化”的闭环流程进行设计，整体采用分层架构，主要包括数据层、数据处理层、模型层、应用层四部分（如图 4-1 至图 4-4 所示）；各层之间通过数据文件、库表与 REST 接口衔接，便于扩展与维护。 
4.3.1 数据层（Data Layer） 
数据层负责数据获取与原始及解析结果的落盘，为后续清洗、建模与分析提供统一来源。数据来源以途家网武汉区域公开页面与接口响应为主，通过自研日历爬虫程序在受控频率下增量更新。采集过程结合接口监听与页面交互，对列表搜索结果进行解析，对详情页日历与标签接口进行捕获；采用随机等待、批次休息与断点续传策略提高稳定性。存储上，解析结果以 JSON 等形式本地持久化并保留备份；随后经导入脚本进入业务库，并可同步至 Hive 原始与明细层，以支持离线聚合与追溯复跑。

4.3.2 数据处理层（Processing Layer） 
数据处理层面向可用数据集与分析指标构建，核心工作包括数据清洗、特征与弱标签构造、Hive 多维离线分析。数据预处理方面，对缺失坐标或关键价格、异常极值等进行规则剔除或截断，对重复房源进行合并或去重，对标签与商圈名称等进行规范化映射。特征工程方面，构造结构、价格、质量、设施与位置等分组特征，设施由关键词规则从标签文本抽取；文本侧采用 TF-IDF 与逻辑回归得到场景等分数，供在线加分使用。Hive 多维分析方面，在数仓汇总层形成行政区、商圈、时间等维度上的均价、分布与热度相关统计，为驾驶舱与论文图表提供指标基础。公开互动量作为热度代理用于展示与对比，不对不可观测成交量作主观还原。

4.3.3 模型层（Model Layer）
模型层将离线指标与特征数据转化为可预测、可推荐、可评估的能力。价格预测采用 XGBoost 回归，输入经统一的特征工程管道，输出建议价格或区间供前端与投资模块使用；训练过程可采用网格搜索或基于验证集的手动调参与交叉验证等方式确定超参数，并以 K 折交叉验证观察稳定性，评价指标包括 MAE、RMSE、MAPE 等。推荐模块离线构建基于内容的相似度矩阵，在线以条件匹配为主、相似度扩展与行为加权为辅。场景分类模块以 TF-IDF 与逻辑回归为轻量文本模型，服务推荐与排序加分。模型文件与矩阵文件由服务启动加载，缺失时推荐策略自动降级。
 

图4-3 模型层架构图
4.3.4 应用层（Application Layer） 
应用层面向用户提供查询、分析、预测、推荐与展示的交互能力。Web 端包括经营驾驶舱、商圈分析、智能定价、个性化推荐、房源列表与详情、我的房源、竞品情报、投资沙盘、价格机会、房源对比、收藏与个人中心等；接口层由 FastAPI 统一暴露，并与鉴权、问卷、浏览历史、收藏等业务逻辑结合。可视化采用 ECharts 等展示趋势、热力、对比与分布。运维上可通过定时任务或手工触发采集与导数脚本更新数据，配合日志与烟测保障版本迭代质量。数据安全方面，仅采集平台公开信息，对用户敏感信息最小化存储与脱敏，接口访问与权限在系统设计范围内予以控制。

通过以上四层设计，系统从公开采集出发，经业务库与 Hive 分析、机器学习与基于内容的推荐，最终以 Web 形态输出商圈洞察、智能定价、价格机会与投资辅助、个性化房源发现等能力，形成与当前工程实现一致、可持续迭代的技术方案。

六、 完成课题所需条件
6.1 数据条件
具备在合规与频控前提下获取途家网武汉区域公开数据的能力；掌握通过 Playwright 进行列表监听、详情日历与标签捕获、断点续传与分文件持久化的采集逻辑。数据存储以采集所得的本地 JSON 等文件与 MySQL 业务库为主，离线分析通过 Hive 完成；Hive 部署于 Docker 容器环境（亦可按需扩展为集群形态），并配备足够磁盘空间与内存，以满足原始快照、业务库表、模型文件存放及 Hive 离线 SQL 执行的需要。
6.2 技术条件
需要具备扎实的大数据处理、浏览器自动化、机器学习与 Web 开发基础，熟练掌握 Python 及 Pandas、XGBoost、Scikit-learn 等库；熟悉 Hive SQL 与离线脚本执行；熟悉 FastAPI 与 React、TypeScript、Ant Design、ECharts 等技术栈；具备前后端联调、接口文档阅读与基本测试能力。硬件方面需要能够运行浏览器自动化、模型训练与前后端服务及 Hive Docker 容器的计算机或服务器环境，并满足容器与宿主机对内存、CPU 与磁盘的配置要求。
七、 存在困难和拟采取的措施
7.1 存在困难
7.1.1 数据采集与还原难度大
主流 OTA平台页面与接口策略可能调整，监听路径或选择器失效会导致采集中断；采集频率与稳定性之间需要长期权衡。数据以挂牌价与公开互动为主，与真实成交存在偏差，预测与价格机会结果应定位为辅助参考。数据使用过程中须遵守法律法规与平台规则，注重最小必要与脱敏存储。
7.1.2 模型适配性与泛化能力挑战
武汉民宿市场具有 “高校刚需 + 商旅游客” 双元结构，不同商圈的价格影响因素权重存在显著差异，单一模型难以适配所有商圈的价格预测需求；民宿房源具有高度非标准化特征，部分小众特色房源的特征数据稀疏，可能导致模型预测精度下降；如何平衡模型复杂度与计算效率，确保模型在实际应用中能快速响应预测需求，也是需要解决的问题。
7.1.3 系统开发与兼容性问题
系统涉及数据采集、处理、分析、可视化等多个模块，不同模块之间的接口设计与协同工作存在一定难度；前端开发需适配不同终端设备，确保响应式布局的兼容性；系统开发完成后，满足课程/实验规模访问，确保系统的响应速度与稳定性，这对系统的性能优化提出了较高要求。
7.2 拟采取的措施
7.2.1 优化数据采集与处理策略，强化合规性管理
针对将监听与解析逻辑模块化，变更时局部修订；充分利用日志、断点续传与备份文件定位故障；对导入与聚合脚本建立可重复执行的检查清单。严格遵守公开数据范围与频控，文档中说明数据来源与局限性；对个人信息坚持最小化与脱敏原则。
7.2.2 改进模型与推荐鲁棒性
在特征与样本划分上关注行政区、商圈分层表现，必要时在论文中分场景讨论误差；对 XGBoost 采用交叉验证与正则化相关设置抑制过拟合。推荐侧明确内容相似与条件匹配主线，矩阵与行为加权作为增强，并在无模型文件时启用热度兜底。通过配置管理 Hive 优先与回退，保证实验可复现。
7.2.3 强化模块化与测试
在后端按领域划分路由与服务，前端按页面与组件拆分，接口契约以后端 OpenAPI 与项目文档为准；开展功能烟测与关键路径手工测试。对驾驶舱、列表、推荐与投资等核心链路保留测试记录，便于论文与答辩演示。发现瓶颈时优先优化 SQL 与缓存策略，而非盲目堆叠复杂度。
